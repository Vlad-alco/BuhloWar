## 2026-07-09 — Фикс: зависание Web-интерфейса (Причина №5 — handleListProfiles безлимитное чтение)

**Задача**
Продолжение исправления периодических зависаний Web-интерфейса. Причина №5: `handleListProfiles()` перебирает все файлы в `/profiles/`, для каждого читает содержимое целиком посимвольно через `content += (char)file.read()`. Нет лимита количества файлов и размера читаемого файла.

**Проблемы:**
1. Посимвольное `String::+=` — тысячи реаллокаций кучи (антифрагментация), заметная пауза при 10+ профилях.
2. Нет лимита файлов — при «мусорной» директории (сотни файлов) функция крутится секунды, блокируя WebServer.
3. Каждый файл читается целиком (бывает 5-10 КБ), хотя нужно только поле `"name"` из начала файла.

**Решение** (3 изменения):

1. **Лимит 50 профилей** — цикл останавливается после 50 `.json` файлов. Остальные пропускаются.
2. **Лимит 2 КБ на файл** — читается только начало файла (имя профиля всегда в начале). Если `"name"` не найдено в первых 2 КБ — используется fallback (имя файла без `.json`).
3. **Буферное чтение по 256 байт** — вместо `content += (char)file.read()` используется `file.read(buf, 256)` + `content.concat(buf, bytesRead)`. Сокращает реаллокации String с тысяч до единиц. `content.reserve(2048)` — единственная аллокация.
4. **Проверка `lock.locked`** — если sdMutex недоступен (500мс таймаут из фикса №4), возвращается 503 вместо работы без мьютекса.
5. **`profilesDir.close()`** — директория явно закрывается после обхода (раньше не закрывалась).

**Изменённые файлы**:
- `AppNetwork.cpp`: `handleListProfiles()` — лимиты, буферное чтение, проверка мьютекса, close директории

## 2026-06-30 — Сессия N+2 (открытая проблема)
### Завышение реальной скорости отбора относительно расчётной после калибровки клапанов

**Задача**:
Зафиксировать проблему — текущий механизм калибровки клапанов даёт значение `capacity` (мл/мин), при котором **реальная скорость отбора в импульсном режиме получается выше расчётной**. Система рассчитывает `openMs`/`closeMs` исходя из заниженной эффективной пропускной способности, фактический отбор идёт быстрее целевого.

**Контекст**:
- Калибровка клапанов запускается автоматически после завершения этапа NASEBYA (см. `ProcessEngine.cpp:931-940` — `if (cfg.calibration) changeStage(Stage::VALVE_CAL)`), то есть при горячей системе и установившемся паровом режиме. Гипотеза о «холодной калибровке» отброшена.
- Текущий алгоритм (`ValveCalMenu.h`): клапан открывается **непрерывно на 60 сек**, пользователь вводит объём в мензурку, `capacity = объём / 1 мин`.
- В реальной работе клапан работает в **импульсном режиме** (openMs/closeMs из `calcValveTiming`, `ProcessEngine.cpp:981`).

**Практические данные**:
- Калиброванное `valve_body_capacity = 120 мл/мин` (7200 мл/ч при 100% открытии).
- Целевая скорость при 2.7 кВт / 600 мл/ч/кВт = 1620 мл/ч.
- Расчётный dutyCycle = 1620 / 7200 = 22.5%.
- По субъективной оценке оператора — реальный отбор **быстрее** целевого. Точный замер мерным стаканом за 1 минуту пока не выполнен.

**Гипотеза причины**:
При непрерывном открытии (калибровка) поток в отборной трубке установившийся, гидростатический напор на клапан минимальный, пропускная способность стационарная.
При импульсной работе (реальный отбор) между импульсами в отборной трубке **накапливается столб жидкости**, который при открытии создаёт дополнительный гидростатический напор → мгновенный расход выше стационарного → средняя скорость отбора выше расчётной.
Чем меньше dutyCycle (медленнее отбор), тем больше накопление и тем сильнее эффект.

**Дополнительные факторы** (менее значимые):
- Инерция столба жидкости в трубке (движущаяся жидкость ускоряется легче, чем старт с нуля при калибровке).
- Разница плотности дистиллята при разной температуре (импульсный режим → теплее → ниже плотность → больше объём при той же массе).
- Избыточное давление паров в дефлегматоре между импульсами → всплеск при открытии.

**Рассмотренные варианты решения** (выбор отложен до получения реальных замеров):

1. **Вариант F — импульсная калибровка**: калибровать клапан не непрерывным открытием, а типичными импульсами openMs/closeMs, которые используются в реальной работе. Накопление столба жидкости учтётся автоматически.
   - Плюсы: точная модель, не требует доп. параметров.
   - Минусы: нужно менять UI мастера калибровки, зависимость от `minOpenTime`.

2. **Вариант C — поправочный коэффициент `dutyFactor > 1`**: ввести множитель в `calcValveTiming`:

   - Плюсы: минимальные изменения кода, низкий риск.
   - Минусы: требует подбора, может отличаться для голов/тела.

3. **Вариант G — мульти-дьюти калибровка**: калибровать на 3 разных dutyCycle (10%, 30%, 60%), интерполировать. Отброшен — слишком сложно, ломает структуру `SystemConfig` и UI.

## 2026-06-11 — Аналитика: архитектура управления мощностью для BuhloWar v3.0

### Исследование замены РМ2М на программное управление мощностью ТЭНа

**Задача**: определить архитектуру управления мощностью для BuhloWar v3.0 — замена ручного регулятора РМ2М (фиксированные, устанавливаемые оператором значения) на систему программного управления, позволяющую ESP32-S3 полностью управлять подаваемой мощностью.

**Контекст**:
В BuhloWar v2.0 оператор вручную устанавливает мощность на РМ2М. Система «видит» температуру через DS18B20, но не может влиять на нагрев. Для v3.0 требуется замкнутый контур: температура → алгоритм → мощность → ТЭН → температура.

**Исходные данные**:
- ТЭН до 40А при 220В (до 8.8 кВт)
- ESP32-S3 DevKitC-1 (N16R8), двухъядерный: Core 0 (сеть), Core 1 (логика)
- Робочий процесс: ректификация — мощность постоянная на всём этапе отбора (головы и тело отбираются при одной мощности)
- Рассмотренный товар: RobotDyn AC Dimmer 40A + плата DIMMERLINK (UART/I2C), AliExpress, производитель rbdimmer.com

**Ключевое исправление понимания процесса ректификации**:
Первоначально была предложена модель с переменной мощностью по этапам (головы 5-15%, тело 25-40%, хвосты 40-60%). Это **ошибочно**. При ректификации мощность поддерживается **постоянной** на протяжении всего процесса отбора. Разделение фракций происходит за счёт изменения скорости/объёма отбора (кран), а не за счёт изменения мощности. Мощность меняется только дважды: разогрев → рабочая мощность.

**Рассмотренные варианты управления мощностью**:

1. **RobotDyn AC Dimmer 40A + DIMMERLINK (UART/I2C)** — фазоимпульсный регулятор:
   - Детектор перехода через ноль (zero-cross) + фазоимпульсное управление симистором (BTA41 или аналог)
   - DIMMERLINK — коммуникационная плата с собственным MCU, снимает с ESP32 задачу точного тайминга
   - Интерфейсы: UART или I2C, команда «установить мощность X%»
   - Каскадирование до ~16 диммеров на одну шину DIMMERLINK
   - Изучена библиотека RBDDimmer (GitHub: RobotDynOfficial/RBDDimmer, 280 stars):
     - Прямое GPIO-управление: 2 пина (ZC input + DIM output)
     - ISR каждые 30 мкс (аппаратный Timer 0 ESP32), прерывание по RISING на ZC пине
     - Функции: `setPower(0-99%)`, `setState(ON/OFF)`, `NORMAL_MODE/TOGGLE_MODE`
     - Проблема для ESP32-S3: ISR каждые 30 мкс — серьёзная нагрузка, конфликт с WiFi на Core 0
   - DIMMERLOGINK решает эту проблему: выносит тайминг на отдельный MCU, ESP32 общается только по UART/I2C
   - Сайт производителя rbdimmer.com заблокирован Cloudflare, документация по протоколу DIMMERLOGINK недоступна

2. **SSR + время-импульсный метод (burst fire)** — включение на N полных периодов из M:
   - Твёрдотельное реле (SSR) с zero-cross — включение только при переходе через ноль
   - Нулевые электромагнитные помехи — полные синусоиды, не обрезанные
   - Минимальная нагрузка на ESP32 — 1 GPIO, никаких ISR и таймеров
   - Для ТЭНа с тепловой инерцией в минуты — разрешение более чем достаточное
   - Окно 2 сек при 50 Гц = 100 полупериодов → шаг 1%

3. **RobotDyn Dimmer + прямое GPIO (без DIMMERLOGINK)**:
   - ISR каждые 30 мкс на ESP32-S3 с WiFi + PSRAM — потенциальная нестабильность
   - Заняты Timer 0 + 2 GPIO — конфликты с другими библиотеками
   - Помехи от фазоимпульсного управления могут влиять на 1-Wire (DS18B20)
   - **Не рекомендуется** для ESP32-S3 с WiFi

**Ключевой вопрос: стабилизация мощности при нестабильном напряжении сети**:
- Burst fire **сам по себе** не стабилизирует мощность — управляет скважностью, но не амплитудой напряжения
- Фазоимпульсный диммер **тоже** не стабилизирует — только обрезает полуволну
- Для стабилизации нужен **замкнутый контур** с измерением реальной мощности, независимо от метода управления
- Датчик: **PZEM-004T v3.0** — измеряет V (80-260В), I (0-100А), P (0-22кВт), PF, частоту, энергию; UART (Modbus-RTU, 9600 бод); встроенная гальваническая развязка; обновление ~200 мс

**Сравнение подходов с обратной связью (PZEM-004T)**:

| Параметр | Burst fire + SSR + PZEM | RobotDyn + DIMMERLOGINK + PZEM |
|---|---|---|
| Скорость реакции | 2-5 сек (достаточно для ТЭНа) | 10-20 мс (избыточно для ТЭНа) |
| Помехи | Нулевые | Фазоимпульсные (влияние на 1-Wire) |
| Нагрузка на ESP32 | Минимальная (1 UART + 1 GPIO) | Минимальная (2 UART) |
| Стоимость | ~$30-45 (SSR + PZEM + реле) | ~$30-45 (Dimmer + DIMMERLOGINK + PZEM) |
| Дополнительные данные | V, I, P, PF, Freq, Energy | V, I, P, PF, Freq, Energy |
| Разрешение | 1% (окно 2 сек) | 1% (100 шагов на полуволну) |

**Принятое решение: Burst fire + SSR (zero-cross) + PZEM-004T v3.0**

Обоснование:
1. Мощность при ректификации — почти константа, плавная регулировка 0-100% нужна только для разогрева и подстройки рабочего значения
2. Нулевые помехи → DS18B20 работает стабильно, не нужны экранирование и ферриты
3. Минимальная нагрузка на ESP32 → нет ISR, нет конфликтов с WiFi и 1-Wire
4. PZEM-004T даёт «бесплатный» вольтметр/амперметр/ваттметр для оператора
5. Скорость реакции 2-5 сек более чем достаточна — инерция ТЭНа и куба на порядки больше
6. Проще, дешевле, надёжнее фазоимпульсного решения

**Архитектура подключения к ESP32-S3**:
- GPIO17/GPIO18 → UART1 (PZEM-004T v3.0, Modbus-RTU)
- GPIO5 → SSR Control (burst fire)
- GPIO6 → Emergency Relay (аварийное отключение ТЭН)
- GPIO3 → 1-Wire (DS18B20 x4) [существующий]
- GPIO8/GPIO9 → I2C (BME280) [существующий]

**Алгоритм стабилизации мощности**:
- П-регулятор с ограничением скорости изменения (не PID)
- Kp ≈ 0.15, max_step = 5% за цикл, период цикла 2 сек
- Причина отказа от PID: интегральная составляющая вызывает перерегулирование при инерционной системе, дифференциальная усиливает шум PZEM
- Чистый П-регулятор + ограничение скорости = стабильное плавное регулирование

**Режимы работы PowerManager**:
| Режим | Описание | Стабилизация |
|---|---|---|
| OFF | ТЭН выключен | Нет |
| RAMP_UP | Плавный разогрев 0→100% за N мин | Нет |
| FULL_POWER | Максимальная мощность (100%) | Нет |
| STABILIZE | Стабилизация мощности | Да (П-регулятор + PZEM) |
| MANUAL | Ручное задание мощности | Опционально |

**Безопасность**:
- Аппаратная: автомат 40А + УЗО 30мА + аварийное реле + биметаллический термостат на кубе (105°C)
- Программная: watchdog PZEM (3 таймаута → emergency OFF), P_max защита, T_куба мониторинг
- SSR безопасное состояние: GPIO LOW = ТЭН выключен (по умолчанию при перезагрузке)

**Интеграция в архитектуру BuhloWar**:
```
Core 0: AppNetwork + CloudManager (WiFi, web, cloud)
Core 1: ProcessEngine + SensorManager + PowerManager ← НОВЫЙ МОДУЛЬ
```

**Дополнительные данные от PZEM-004T для оператора**:

## 2026-06-11 — Аналитика: сравнение датчиков температуры DS18B20 vs Pt100 + алгоритмы обнаружения шпоры

### Исследование применимости DS18B20 и Pt100 в системе BuhloWar v2.0

**Задача**: провести детальное сравнение текущих датчиков DS18B20 (4 шт., 1-Wire, GPIO3) и платиновых термометров сопротивления Pt100 для принятия решения о целесообразности замены.

**Контекст**:
Система BuhloWar v2.0 использует 4 датчика DS18B20 (TSA, AQUA, TSAR, TANK) на единой 1-Wire шине и BME280 для атмосферного давления. Ключевые температурные параметры: hysteresis = 0.24°C, delta (шпора) = 0.06°C (параметр), минимальное рабочее значение delta = 0.12°C.

**Результаты сравнения**:

1. **Точность**:
   - DS18B20: ±0.5°C во всём диапазоне
   - Pt100 кл. A: ±0.15°C при 0°C, ±0.35°C при 100°C (в 1.4-1.6x лучше в диапазоне 78-100°C)
   - Pt100 кл. B: ±0.30°C при 0°C, ±0.80°C при 100°C (существенно хуже кл. A при высоких T)

2. **Дистилляция**: DS18B20 полностью достаточен — пороги разделены интервалами 7-14°C, погрешность 0.5°C не влияет на управление процессом. Основная неопределённость от тепловой инерции системы (30-60 сек), а не от датчика.

3. **Ректификация**: Pt100 кл. A даёт преимущество в расчёте крепости (±0.3-0.7% об. вместо ±1-2% об. при погрешности DS18B20 0.5°C), но для обнаружения шпоры преимущество минимально (см. п.5).

4. **Стоимость замены**:
   - DS18B20 (4 шт.): ~420 руб.
   - Pt100 + 4x MAX31865: ~6200-8200 руб. (нужны 4 SPI-чипа, 16 GPIO)
   - Pt100 + ADS124S08: ~5400-7400 руб. (1 АЦП, 4 GPIO, но сложнее ПО)
   - Разница: 13-20x, плюс переработка PCB и 60-120 часов разработки ПО

5. **Ключевой инсайт — относительная vs абсолютная точность**:
   - Для управления процессом (шпора, гистерезис, пороги) важна **относительная** точность — способность обнаруживать *изменения* температуры, а не абсолютное значение
   - DS18B20 с разрешением 0.0625°C и краткосрочной стабильностью ±1 LSB надёжно отслеживает изменения
   - При delta = 0.12°C (минимальное реальное рабочее значение) пороговый метод уже работает надёжно — шум ±0.0625°C не может дать ложное срабатывание на 0.12°C

6. **Алгоритм обнаружения шпоры по производной**:
   - Рассмотрен метод: вместо «температура выше порога?» спрашивать «растёт ли температура?»
   - dT/dt при стабильной ректификации ≈ 0 (шум с нулевым средним)
   - dT/dt при шпоре > 0 (устойчивый рост)
   - Метод эффективен при delta = 0.06°C (1 LSB), где шум даёт ложные срабатывания порогового метода
   - **При delta = 0.12°C (2 LSB) метод производной не даёт значимого преимущества** — пороговый метод уже работает надёжно, шум физически не может дать скачок 0.12°C

7. **Калибровка DS18B20** (без замены аппаратуры):
   - Одноточечная: лёд+вода (0°C) → offset → погрешность ±0.2-0.3°C
   - Двухточечная: лёд+вода (0°C) + кипящая вода (100°C) → offset + gain → погрешность ±0.1-0.2°C
   - Трёхточечная: + кипящий спирт (~78°C) → кусочно-линейная → ±0.05-0.15°C в рабочем диапазоне
   - Хранение: 4 датчика × 2 float (offset+gain) = 36 байт в EEPROM

**Выводы и рекомендации**:

1. **DS18B20 оставляем** — для дистилляции точность достаточна, для ректификации при delta=0.12°C — тоже
2. **Калибровка DS18B20** — имеет смысл только для расчёта крепости (абсолютная точность), для управления процессом не нужна
3. **Pt100** — рассматривать только для BuhloWar v3.0 как опцию для TSAR (один канал), остальное DS18B20
4. **Метод производной** — не актуален при delta=0.12°C, пороговый метод работает надёжно

**Сгенерированные файлы**:
- `docs/DS18B20_vs_Pt100_BuhloWar.pdf` — полный аналитический отчёт (13 стр., 7 таблиц)
- `download/generate_sensor_comparison.py` — скрипт генерации PDF

## 2026-05-31 — Фикс: CloudManager не инициализируется при фоновом переходе AP→STA + LCD не обновляет символ сети

### BuhloWar.ino + AppNetwork.h + AppNetwork.cpp — CloudManager init + LCD network symbol

**Задача**: при фоновом переходе AP→STA (WiFi подключился в фоне) CloudManager не инициализируется, телеметрия не отправляется, символ сети на LCD не меняется с 'A' на 'W'.

**Анализ корневой причины**:

1. **CloudManager не инициализируется** — в `setup()` CloudManager создаётся только при `netMode == STA_MODE`. При фоновом переходе AP→STA в `update()` этот код не выполняется повторно. `cloudManager.isConfigured()` возвращает false → телеметрия не шлётся.

2. **LCD символ не обновляется** — `processEngine.updateNetworkStatus()` вызывается только внутри `if (processEngine.isProcessRunning())`. В idle режиме (процесс не запущен) символ 'A' на LCD не меняется на 'W' даже после перехода в STA_MODE.

**Решение** (3 изменения):

1. **Флаг `didSwitchToSTA()`** — в AppNetwork добавлен флаг `switchedToSTA`, который устанавливается при переходе AP→STA в `update()`. Метод `didSwitchToSTA()` возвращает true один раз и сбрасывает флаг. Аналог edge-trigger.

2. **Инициализация CloudManager в loop()** — при обнаружении `didSwitchToSTA()` в основном цикле, CloudManager инициализируется с теми же параметрами (URL, API key, command callback), что и в `setup()`. Защита от повторной инициализации: `!cloudManager.isConfigured()`.

3. **Обновление LCD символа сети всегда** — добавлен блок в `loop()` который отслеживает изменение символа сети (`W`/`A`/`X`) через static-переменную `lastNetSymbol`. При изменении — вызывает `processEngine.updateNetworkStatus()` независимо от того, запущен процесс или нет.

**Изменённые файлы**:
- `AppNetwork.h`: добавлен метод `didSwitchToSTA()` и поле `switchedToSTA`
- `AppNetwork.cpp`: реализация `didSwitchToSTA()`, установка `switchedToSTA=true` при AP→STA переходе
- `BuhloWar.ino`: блок инициализации CloudManager при `didSwitchToSTA()`, обновление LCD символа сети в idle

## 2026-05-31 — Фикс: Kernel Panic — WiFi.disconnect(true) убивает AP режим + переполнение стека CloudManager

### AppNetwork.cpp + CloudManager.h — безопасный WiFi disconnect + увеличенный стек + мониторинг

**Задача**: система падает с Kernel Panic через 5-8 минут после запуска. WiFi подключается, Cloud работает, затем краш и перезагрузка. Циклично.

**Анализ корневой причины**:

1. **WiFi.disconnect(true) убивает AP режим** — предыдущий фикс добавил `WiFi.disconnect(true)` перед `WiFi.begin()`, но `disconnect(true)` полностью выключает WiFi радио (WIFI_OFF). К этому моменту AP режим уже запущен через `startWebServerEarly()` — WebServer и DNS сервер привязаны к AP интерфейсу. Когда WiFi выключается, эти объекты остаются привязаны к несуществующему интерфейсу → Kernel Panic при обращении к ним.

2. **Переполнение стека CloudManager** — cloud задача запущена со стеком 8KB, но SSL/TLS требует 6-10KB стека, а JSON парсинг и буферы ещё ~4KB. При большом JSON телеметрии стек переполняется → Kernel Panic.

**Решение** (3 изменения):

1. **Безопасный WiFi disconnect** — `WiFi.disconnect(false)` вместо `disconnect(true)`. Параметр `false` отключается от STA точки доступа, но НЕ выключает WiFi радио. AP режим остаётся живым. Дополнительно `WiFi.mode(WIFI_AP_STA)` гарантирует dual режим.

2. **Увеличенный стек CloudManager** — 16KB вместо 8KB. SSL/TLS требует 6-10KB, JSON + буферы ещё ~4KB. 16KB даёт безопасный запас.

3. **Мониторинг стека в CloudManager** — каждые 60 сек логируется свободный остаток стека (`uxTaskGetStackHighWaterMark`). При остатке < 2KB — предупреждение. Помогает диагностировать переполнение до Kernel Panic.

**Изменённые файлы**:
- `AppNetwork.cpp`: `WiFi.disconnect(true)` → `WiFi.disconnect(false)` + `WiFi.mode(WIFI_AP_STA)`
- `CloudManager.h`: стек 8192 → 16384, мониторинг стека каждые 60 сек

## 2026-05-31 — Фикс: система не переходит из AP в STA режим при подключении WiFi

### AppNetwork.cpp — обнаружение фонового подключения WiFi + чистый старт + увеличенный таймаут

**Задача**: при подключении к другой точке WiFi (данные на SD карте) система получает IP от роутера, веб-интерфейс доступен по этому IP, но система показывает режим "A" (AP_MODE) вместо "W" (STA_MODE). В результате телеметрия не отправляется на хостинг.

**Анализ корневой причины**:
1. `startWebServerEarly()` устанавливает `networkMode = AP_MODE`
2. `begin()` пытается подключиться к WiFi с таймаутом 9 сек, но при подключении к новой AP этого времени может не хватить
3. `networkMode` остаётся `AP_MODE`
4. ESP32 в фоне продолжает попытки и подключается к роутеру (поэтому IP есть)
5. В `update()` проверка `if (networkMode == STA_MODE)` — false, код внутрь не заходит
6. Режим навсегда остаётся AP_MODE, хотя WiFi реально подключён

**Решение** (4 изменения):

1. **WiFi.disconnect(true) перед WiFi.begin()** — при смене точки доступа WiFi может "застрять" на старых креденшиалах. `disconnect(true)` очищает кэш WiFi и позволяет чисто подключиться к новой AP. Вызывается перед каждой попыткой `WiFi.begin()` в `begin()`.

2. **Увеличенный таймаут в begin()** — 3 попытки × 5 сек = 15 сек максимум (было 3×3=9 сек). При подключении к новой AP ESP32 может требовать больше времени.

3. **Обнаружение фонового подключения (AP→STA)** — добавлена проверка в `update()`: если `networkMode == AP_MODE`, но `WiFi.status() == WL_CONNECTED` — переключаемся в STA_MODE, проверяем интернет, синхронизируем NTP. Это ключевое исправление: без него система навсегда оставалась в AP_MODE.

4. **Ускоренная проверка в AP режиме** — интервал периодической проверки сети в AP_MODE уменьшен с 30 до 10 сек, чтобы быстрее обнаружить подключение WiFi в фоне и переключиться в STA_MODE.

**Изменённые файлы**:
- `AppNetwork.cpp`:
  - `begin()`: WiFi.disconnect(true) + delay(100) перед WiFi.begin(), tries < 50 вместо < 30
  - `update()`: блок перехода AP→STA при WiFi.status() == WL_CONNECTED
  - `update()`: safeInterval = 10000 для AP_MODE (было 30000 единый для всех)

### CloudManager.h — FreeRTOS задача: нулевая блокировка loop
`0ccf4ae`

- **Все HTTP-запросы вынесены в отдельную FreeRTOS задачу на core 0** (там WiFi/TCP стек). Основной loop на core 1 больше никогда не блокируется, независимо от состояния сервера.
- **Архитектура**: cloud задача (core 0, приоритет 0, стек 8KB) выполняет все HTTP-запросы, backoff, управление `_client`. Основной loop вызывает `update()` — который копирует телеметрию через mutex (~2 мкс), читает очереди команд/настроек, вызывает callbacks. Задержка — ноль.
- **Связь loop↔cloud**: телеметрия — mutex (запись из loop, чтение из cloud); команды и настройки — FreeRTOS очереди (cloud→loop, неблокирующее чтение).
- **Callbacks** `onCommand`/`onSettings` вызываются из контекста основного loop — безопасно, без race conditions.
- **Ресурсы**: ~16KB heap (8KB стек задачи + 7KB очереди + 1KB overhead).
- Публичный API идентичен — **никаких изменений в других файлах**.

### CloudManager.h — _client.stop() только при ошибках
`8d9caea`

- Предыдущая версия вызывала `_client.stop()` перед каждым запросом, что заставляло делать новый TLS handshake каждый раз (2-5 сек overhead). Web-интерфейс с SD карты начал притормаживать, система зависала.
- Теперь `_client.stop()` вызывается **только при `errorCount > 0`** (предыдущий запрос завершился ошибкой = мёртвое TCP соединение). При успехе соединение переиспользуется с нулевой задержкой.

### CloudManager.h — Предотвращение 3-минутного зависания loop
`c302901`

- **Причина**: при недоступности сервера TCP ретрансмиссии на переиспользуемом мёртвом соединении блокировали loop на 1→2→4→8→16 сек суммарно до 3 минут.

### Документация: REF TSAR/REF АД и коррекция давления в TELO
Вопросы при тестировании Стандарт метода:

- **REF TSAR** (`rtsarM`) и **REF АД** (`adPressM`) в WEB мониторинге — это расчётные значения, зафиксированные **один раз** при входе в TELO (`rtsarM < 0.1`). Не меняются весь процесс, ни при залётах, ни при переходах NASEBYA → TELO. Обнуляются только при сбросе/начале новой колонны. Одинаковы для Стандарт и Шпора.
- **Коррекция давления** `(data.pressure - adPressM) * 0.0278` — применяется в обоих методах:
  - **Шпора**: учитывается для delta (снижение скорости) и hysteresis (залёт).
  - **Стандарт**: учитывается только для hysteresis (залёт → NASEBYA или finishTelo). Delta нет, так как в Стандарте нет механизма плавного снижения скорости — только бинарное решение "залёт / не залёт".
  - При росте давления (фронт с низким уходит): пороги повышаются → залёт позже. При падении: пороги снижаются → залёт раньше.
